Solution dedicated for mobile
You:
  • Need to use any phone accessories like camera
  • Want to send push notifications
  • Require 'native' feel on every phone OS
  • Are confident that user wants to install your app
And you are you ready to pay a lot for that
You want an application which looks nicely on mobile and rearranges itself depending on the size of the screen, but the functionality needs to be exactly the same as on desktop.
You don't want basic web application - that's fine. You still have a chance to save some money by writing major part of your code only once and re-using it on all the plaftorms. There are two considerable ways to go.
You're rich and you want native apps for all the platforms. If you want to hit a mass market, most probably you will develop solutions at least for Android and for iOS. Native application can meet all the platform design requirements and navigation specifics, without any 'middleman' overhead.
Get traditional web-app (second section) and make sure to hire good web designers who will make it responsive. And you're welcome for saving you lots of money.
Android is definitely the most closed open-source project in the world. But to code on this platform you will need some Java skills + experience.
For iOS you will need Apple Swift magicians - gossips claim that Swift is pretty awesome so you don't want to go for Objective C solution - which is an earlier standard.
For Windows Phone you would need XAML and C#. Have in mind that even Snapchat didn't bother to to code for Windows Phone, so there are very few situations in which you should do so. I'm not even mentioning Blackberry, since I'd rather develop for Xiaomi than for it.
Apache Cordova (widely recognized in Phonegap distribution). Mentioning it makes me feel a little bit guilty, since it may seem that I'm trying to trick you into browser-based solution again. Yes, that's true that Cordova actually runs in a browser, but it's very unlikely that your user will realize that! Check out some implementations to see it yourself. It's huge advantage it's simplicity - solution can be delivered using exclusively JS, HTML and CSS. I had a pleasure to test the plugin, which enables to develop for mobile using AngularJS. It was extremely easy to get started and the development proceeded surprisingly, fast as long as I was using very typical use cases. When it came to using some plugins it took me ages to discover that relatively popular one wasn't yet working with newest software version and errors weren't my configuration's fault. It has been 2 years ago and the community has grown since then a lot since then, so that argument most probably isn't valid anymore. If you're considered about the future of the framework - well, you never know, but Cordova is backed by Adobe, who donated it to Apache Foundation.
React Native enables you to write applications using ReactJS (which most front-end developers are already familiar with) but encorporating some native components of each mobile operation systems, like tab bar in iOS and DrawerLayout on Android. If you don't believe that, you might want to check those apps out. Solution seems to be extremely handy. Additionally blessing from Zuckerberg's corporation increases chances that the support for it won't stop after community finds new hype topic.

Haml and Jade (which actually derived from Haml) are markup languages, simplifying HTML syntax and replacing inline page template engines. When templates are useful? Whenever you want to display some server-calculated values inside of well formatted HTML. For example if you want to send a nicely formatted email to many users and but include some personalised data - like their username or purchases - use their name and list of their purchases - you might want to use templates.
There are rumors that Haml and Jade can be useful on some other occasions - I haven't been given to experience that, yet.

JS Pre-processors

Coffeescript is a simple pre-processor, which enables you to write js code faster. Instead of curved brackets you use proper indentation, you can omit 'function' and 'return' statements and easily iterate ever an array. 99% times it'll save you from some small work, but when you'll have problem with it, you might struggle for ages to spot it. Anyway, statistically everyone in my team loved it. I was a statistical error.

I understand that immutability for some people can seem over the top, but strong typing is a baseline. So how to live in the world of JavaScript?
Typescript is a typed superset of JavaScript that compiles to plain JavaScript. How cool is that? And I'm not done yet. It also adds classes, constructors, interfaces... if you like that stuff and rather handle compile-time errors, that's good stuff for you.

Angular used to be my favourite one. And it's very clear why - with AngularJs you write services and controllers - just like on the backend! It's especially good for data binding - you can easily display the variables and even loop through the arrays as well as use conditional statements connecting your js-code to the interface. That comes with a price - it's quite resource-consuming, but unless you will use hundreds of permanently watched objects, you shouldn't worry too much. Remember that Angular 2.0 will not be backward compatible!

I struggle to trust anything that mixes the syntax with HTML, but ReactJs' community grows incredibly fast and it's very improbable that so many smart people are wrong - so let's have a closer look. With React, instead of separating the technologies (HTML, CSS and JS), we separate whole logical entities instead, grouping it and encapsulating inside components. There is no good, old MVC, but an architecture called Flux (or going further Reflux or Redux). Zuckerberg uses Flux, so this is what I will discuss. User and API's communicate with Actions - those are simple helpers passing info to dispatchers. Dispatchers broadcast whatever is needed to registered callbacks (stored in Store) which can communicate back with the view. That's not the place to go into the details, but rather to get your attention. If it got also your interest, you might want to read this
Ah, and worth mentioning - React doesn't have it's own router. You might use a third-party module for that, though.
